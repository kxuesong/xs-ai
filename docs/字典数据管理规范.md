# 字典数据管理规范

> 本文档定义业务单据类型、状态等常量的字典配置规范

## 一、字典系统概述

### 1.1 什么是字典

字典是用于管理系统中**枚举类型数据**的配置系统，包括：
- 业务单据类型（如：订单类型、支付方式）
- 业务状态（如：启用/禁用、成功/失败）
- 选项配置（如：性别、通知类型）

### 1.2 为什么使用字典

**❌ 硬编码方式的问题：**
```java
// 硬编码在代码中
if ("0".equals(status)) {
    return "正常";
} else if ("1".equals(status)) {
    return "停用";
}
```

**✅ 字典方式的优势：**
- **可配置**：运维人员可通过后台管理界面修改，无需改代码
- **统一管理**：所有枚举数据集中在数据库，便于维护
- **国际化支持**：可根据语言返回不同标签
- **前后端共享**：前后端使用同一套字典数据
- **缓存优化**：自动缓存，高性能

### 1.3 字典数据表结构

#### sys_dict_type（字典类型表）

```sql
CREATE TABLE sys_dict_type (
    dict_id     BIGINT       PRIMARY KEY COMMENT '字典主键',
    dict_name   VARCHAR(100) NOT NULL COMMENT '字典名称',
    dict_type   VARCHAR(100) NOT NULL UNIQUE COMMENT '字典类型',
    status      CHAR(1)      DEFAULT '0' COMMENT '状态（0正常 1停用）',
    remark      VARCHAR(500) COMMENT '备注',
    create_time DATETIME     COMMENT '创建时间',
    update_time DATETIME     COMMENT '更新时间'
) COMMENT='字典类型表';
```

**示例数据：**
| dict_id | dict_name | dict_type | status |
|---------|-----------|-----------|--------|
| 1 | 系统状态 | sys_common_status | 0 |
| 2 | 用户性别 | sys_user_sex | 0 |
| 3 | 支付状态 | pay_state | 0 |

#### sys_dict_data（字典数据表）

```sql
CREATE TABLE sys_dict_data (
    dict_code   BIGINT       PRIMARY KEY COMMENT '字典编码',
    dict_sort   INT          DEFAULT 0 COMMENT '字典排序',
    dict_label  VARCHAR(100) NOT NULL COMMENT '字典标签',
    dict_value  VARCHAR(100) NOT NULL COMMENT '字典键值',
    dict_type   VARCHAR(100) NOT NULL COMMENT '字典类型',
    css_class   VARCHAR(100) COMMENT '样式属性（CSS类名）',
    list_class  VARCHAR(100) COMMENT '表格回显样式',
    is_default  CHAR(1)      DEFAULT 'N' COMMENT '是否默认（Y是 N否）',
    status      CHAR(1)      DEFAULT '0' COMMENT '状态（0正常 1停用）',
    remark      VARCHAR(500) COMMENT '备注',
    create_time DATETIME     COMMENT '创建时间'
) COMMENT='字典数据表';

CREATE INDEX idx_dict_type ON sys_dict_data(dict_type);
```

**示例数据：**
| dict_code | dict_type | dict_label | dict_value | list_class |
|-----------|-----------|-----------|-----------|-----------|
| 1 | sys_common_status | 正常 | 0 | primary |
| 2 | sys_common_status | 停用 | 1 | danger |
| 3 | sys_user_sex | 男 | 0 | |
| 4 | sys_user_sex | 女 | 1 | |

## 二、字典命名规范

### 2.1 字典类型命名（dict_type）

**格式：** `{模块}_{业务}_{类型}`

**规则：**
- 使用小写字母和下划线
- 前缀表示模块（sys=系统、pay=支付、chat=聊天）
- 中间表示业务对象
- 后缀表示类型（status=状态、type=类型、category=分类）

**示例：**
```
sys_common_status      # 系统通用状态
sys_user_sex           # 用户性别
sys_notice_type        # 通知类型
pay_state              # 支付状态
pay_method             # 支付方式
chat_model_category    # 聊天模型分类
wf_business_status     # 工作流业务状态
```

### 2.2 字典值命名（dict_value）

**规则：**
- **数字型**：推荐使用 0、1、2... 表示状态或类型
- **字符串型**：使用大写字母和下划线（如：WECHAT_PAY）
- **布尔型**：使用 Y/N 或 0/1

**示例：**
```java
// 状态类 - 使用数字
0: 正常/启用/成功
1: 停用/禁用/失败
2: 其他状态

// 类型类 - 使用字符串或数字
ALIPAY、WECHAT_PAY、BANK_CARD
或 0、1、2

// 布尔类 - 使用 Y/N 或 0/1
Y/N、0/1
```

### 2.3 字典标签命名（dict_label）

**规则：**
- 使用中文表示（支持国际化时可扩展）
- 简洁明了，一般 2-4 个字
- 与业务语境一致

**示例：**
```
正常、停用、启用、成功、失败
男、女、未知
待支付、已支付、已退款
```

## 三、后端使用规范

### 3.1 查询字典数据

```java
@Service
@RequiredArgsConstructor
public class MyBusinessService {

    private final ISysDictTypeService dictService;  // 实现了 DictService 接口

    public void businessMethod() {
        // 方法1：根据字典值获取标签（推荐）
        String statusLabel = dictService.getDictLabel("sys_common_status", "0");
        // 返回: "正常"

        // 方法2：批量获取标签（支持分隔符）
        String sexLabels = dictService.getDictLabel("sys_user_sex", "0,1", ",");
        // 返回: "男,女"

        // 方法3：根据标签获取值（逆向查询）
        String statusValue = dictService.getDictValue("sys_common_status", "正常");
        // 返回: "0"
    }
}
```

### 3.2 在实体类中使用字典

```java
@Data
@TableName("chat_session")
public class ChatSession extends BaseEntity {

    /**
     * 会话状态
     * 字典类型: sys_common_status
     * 0=正常 1=停用
     */
    private String status;

    /**
     * 模型分类
     * 字典类型: chat_model_category
     * OPENAI, DEEPSEEK, ZHIPU
     */
    private String modelCategory;
}
```

**注释规范：**
- 必须注明字典类型（dict_type）
- 列举主要的字典值和含义
- 便于后续维护

### 3.3 在 VO 中转换字典

**使用注解自动转换（如果支持）：**
```java

```

### 3.5 缓存机制说明

字典数据使用**三层缓存**：

```java
// L1: 请求级缓存（最快）
// 使用 SaHolder.getStorage() 存储，生命周期为单次请求
public String getDictLabel(String dictType, String dictValue) {
    List<SysDictDataVo> datas = (List<SysDictDataVo>)
        SaHolder.getStorage().get(CacheConstants.SYS_DICT_KEY + dictType);

    if (ObjectUtil.isNull(datas)) {
        // L2: Redis缓存
        datas = this.selectDictDataByType(dictType);
        // 存入请求级缓存
        SaHolder.getStorage().set(CacheConstants.SYS_DICT_KEY + dictType, datas);
    }
    // ...
}

// L2: Redis缓存（推荐）
@Cacheable(cacheNames = CacheNames.SYS_DICT, key = "#dictType")
public List<SysDictDataVo> selectDictDataByType(String dictType) {
    // L3: 数据库查询
    return dictDataMapper.selectDictDataByType(dictType);
}
```

**缓存更新策略：**
- 新增/修改字典数据 → 自动更新 Redis 缓存
- 删除字典数据 → 自动清除 Redis 缓存
- 管理员刷新缓存 → 清空所有字典缓存

## 六、最佳实践

### 6.1 命名规范

✅ **好的命名：**
```
sys_common_status      # 清晰明了
pay_state              # 简洁
chat_model_category    # 含义明确
```

❌ **差的命名：**
```
status                 # 太泛化，不知道是什么状态
dict1                  # 无意义
userSex                # 应使用下划线
```

### 6.2 何时使用字典

**✅ 适合使用字典：**
- 枚举值较少（< 50个）
- 值相对稳定，不频繁变化
- 需要前后端共享
- 需要界面化管理

**❌ 不适合使用字典：**
- 数据量大（如省市区数据）
- 频繁变化的业务数据
- 需要关联其他表的数据
- 复杂的层级结构

**替代方案：**
```java
// 大量数据 → 独立业务表
CREATE TABLE sys_area (
    area_id INT PRIMARY KEY,
    area_name VARCHAR(100),
    parent_id INT
);

// 频繁变化 → 业务配置表
CREATE TABLE biz_config (
    config_key VARCHAR(100),
    config_value TEXT
);
```

### 6.3 字典值设计原则

**原则1：使用数字表示状态**
```java
// ✅ 推荐
0: 正常
1: 停用
2: 删除

// ❌ 不推荐（字符串占用空间大）
NORMAL: 正常
DISABLED: 停用
```

**原则2：预留扩展空间**
```java
// 订单状态
0: 待支付
1: 已支付
2: 已发货
3: 已完成
// 预留 4-9 供将来扩展
10: 已取消
11: 已退款
```

**原则3：避免重复值**
```sql
-- ❌ 错误：同一字典类型中值重复
INSERT INTO sys_dict_data (dict_type, dict_label, dict_value) VALUES
('order_status', '待支付', '0'),
('order_status', '待确认', '0');  -- 值重复！

-- ✅ 正确：每个值唯一
INSERT INTO sys_dict_data (dict_type, dict_label, dict_value) VALUES
('order_status', '待支付', '0'),
('order_status', '待确认', '1');
```

### 6.4 性能优化

**1. 批量加载字典**
```typescript
// ✅ 一次性加载多个字典
Promise.all([
  getDictOptions(DictEnum.SYS_COMMON_STATUS),
  getDictOptions(DictEnum.SYS_USER_SEX),
  getDictOptions(DictEnum.PAY_STATE),
]);

// ❌ 多次独立加载（效率低）
getDictOptions(DictEnum.SYS_COMMON_STATUS);
getDictOptions(DictEnum.SYS_USER_SEX);
getDictOptions(DictEnum.PAY_STATE);
```

**2. 避免循环中查询字典**
```java
// ❌ 错误：每次循环都查询字典
for (Order order : orders) {
    String label = dictService.getDictLabel("order_status", order.getStatus());
    // ...
}

// ✅ 正确：先查询一次，然后使用Map
List<SysDictDataVo> dictList = dictService.selectDictDataByType("order_status");
Map<String, String> dictMap = StreamUtils.toMap(
    dictList,
    SysDictDataVo::getDictValue,
    SysDictDataVo::getDictLabel
);

for (Order order : orders) {
    String label = dictMap.get(order.getStatus());
    // ...
}
```

## 七、常见问题

### Q1: 字典缓存何时更新？

**A:** 自动更新场景：
- 新增字典数据 → 自动更新该类型的缓存
- 修改字典数据 → 自动更新该类型的缓存
- 删除字典数据 → 自动清除该类型的缓存

**手动刷新：**
```bash
# 调用API刷新缓存
POST /system/dict/type/refreshCache
```

### Q2: 前端如何强制刷新字典？

**A:** 清除 Store 缓存
```typescript
import { useDictStore } from '#/store/dict';

const dictStore = useDictStore();
dictStore.resetCache();  // 清除所有字典缓存
```

### Q3: 字典值可以修改吗？

**A:** ⚠️ **不推荐修改 dict_value**

原因：
- dict_value 已保存在业务表中
- 修改后会导致数据不一致

**正确做法：**
- 只修改 dict_label（显示文本）
- 只修改 list_class（样式）
- 如需修改值，必须同步更新所有关联业务数据

### Q4: 如何支持国际化？

**A:** 扩展字典表结构
```sql
ALTER TABLE sys_dict_data ADD COLUMN dict_label_en VARCHAR(100) COMMENT '英文标签';

-- 或创建独立的国际化表
CREATE TABLE sys_dict_data_i18n (
    dict_code BIGINT,
    locale VARCHAR(10),  -- zh_CN, en_US
    dict_label VARCHAR(100),
    PRIMARY KEY (dict_code, locale)
);
```

## 八、总结

### 核心要点

1. **统一使用字典管理枚举数据**，避免硬编码
2. **遵循命名规范**：`{模块}_{业务}_{类型}`
3. **使用常量类和枚举**，提高代码可维护性
4. **善用缓存机制**，提升性能
5. **定期维护字典数据**，保持数据一致性

### 检查清单

在开发新功能时，如果涉及枚举数据，请检查：

- [ ] 是否新增了字典类型（sys_dict_type）
- [ ] 是否新增了字典数据（sys_dict_data）
- [ ] 是否在后端常量类中添加了字典类型
- [ ] 是否在前端枚举中添加了字典类型
- [ ] 是否在实体类中添加了字典注释
- [ ] 是否配置了合适的 list_class 样式
- [ ] 是否测试了缓存刷新功能
